Design rationale

The current HTML packs ~200 fields per community into a single nested JavaScript object — a god entity. The gold layer decomposes it into a star schema: 3 shared dimensions and 17 fact tables split into point-in-time snapshots (one row per community, wide format) for static dashboard fields, and time-series tables (long format) for trend charts and historical comparisons. Snapshot tables map 1:1 to dashboard sections. Time-series tables are normalized so each row carries one observation — no pivoted year columns like ei2016, ei2017, ei2018.... The HTML build step pivots these back into the communityData{} object at render time.

Shared dimension tables referenced by all fact tables via foreign key. These are the backbone of the star schema — every fact row points back here.

Business rules for null handling per your gold benchmark criteria. Every table defaults to: NULLs on FK columns = referential integrity failure. NULLs on measure columns follow table-specific rules below.

All snapshot tables
No NULLs on geography_id or source_id. NULL indicator_values are suppressed rows — keep NULL, flag with companion _is_suppressed column only where Census suppression applies.
cmhc_rental_market
NULL for rent_bachelor where CMHC suppresses (small sample). Vacancy NULL for years without CMHC survey coverage.
hna_housing_deficit
Zero-fill: deficit_count = 0 means no deficit in that cell, not missing data.
hna_building_permits
NULL for permit_value in years where only counts are available. NULL for entire rows where community didn't report.
hna_priority_groups
Zero-fill for suppressed small counts (count=0, rate=0.0) per HART convention.
